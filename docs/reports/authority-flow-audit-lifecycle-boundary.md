## 2. Authority Map

### 2.1 DB writes (`agent_launch_registry`)

| Writer | Path | Authority | Coordinated? |
|--------|------|-----------|-------------|
| `claim()` via `request_launch()` | CLI → Service → Repository | **Authoritative** (INSERT) | CAS + UNIQUE constraint |
| `cas_update_status()` via service methods | CLI → Service → Repository | **Authoritative** (UPDATE) | CAS guard on status+version |
| `reconcile_expired_leases()` | Not called from skill | **Delegated** (cleanup) | No caller currently |

**Verdict:** Single-writer to DB. All writes go through `SqliteAgentLaunchRepository`. Skill NEVER writes directly. ✅ ADR-002 enforced.

### 2.2 File writes (skill /tmp state)

| Artifact | Writer 1 | Writer 2 | Coordinated? |
|----------|----------|----------|-------------|
| `.meta` (JSON) | `register()` — initial write | `done-helper.sh` — update exitCode/stopReason | **Sequential** (register before done) ✅ |
| `.done` (JSON) | `done-helper.sh` — normal exit | `trap-helper.sh` — SIGTERM | **Exclusive** (one fires per process) ✅ |
| `.pane` (text) | `register()` — pane index | `unregister()` — delete | **Sequential** ✅ |
| `.done/.meta/.pane` | `kill-all` — bulk delete | — | **Deletes without lifecycle** ⚠️ |

### 2.3 `/tmp/fork-live-registry/` vs `REG_DIR`

| Location | Set by | Purpose |
|----------|--------|---------|
| `REG_DIR` (in core.sh) | `cmd_init` | `.pane`, `.meta`, `.done` files |
| `/tmp/fork-live-registry/` | tmux-live monolith (legacy) | `.meta` files (duplicate) |
| `/tmp/fork-live-done/` | tmux-live monolith (legacy) | `.done` files (duplicate) |

**Potential split-brain:** `kill-all` cleans `REG_DIR` files but does NOT clean `/tmp/fork-live-registry/` or `/tmp/fork-live-done/`.

## 3. Findings

### F1: `kill`/`kill-all` bypass lifecycle — launches orphaned in DB ⚠️ HIGH

**Severity:** HIGH
**Confidence:** high (direct-write confirmed)
**Heuristic:** H3 — Hook mutates state outside lifecycle

**Evidence:**
- `cmd_kill()` sends SIGKILL, kills pane, calls `unregister()`. Does NOT call `mark-failed` or `begin-termination`.
- `cmd_kill_all()` bulk-deletes `.pane`/`.done`/`.meta` files. Does NOT iterate agents to call lifecycle transitions.
- `cmd_stop()` sends SIGTERM, waits for done marker. If done-helper fires, lifecycle is called. If it escalates to `cmd_kill`, lifecycle is bypassed.

**Impact:** After `kill` or `kill-all`, launches remain ACTIVE/SPAWNING in DB. Only cleaned by `reconcile_expired_leases()` after 5-minute lease expiry.

**Recommendation:** Wire lifecycle into kill path. Before killing, read `.meta` to get `launch_id`, call `mark-failed` with error "Killed by operator". For `kill-all`, iterate all registered agents before bulk cleanup.

---

### F2: `lifecycle_get_cli_base` has asymmetric fallback ⚠️ MEDIUM

**Severity:** MEDIUM
**Confidence:** high (code diff confirmed)
**Heuristic:** H1 — Multiple paths, different behavior

**Evidence:**
- `_call_fork_launch()` — if no venv AND no global `fork` → returns 1 (hard fail)
- `lifecycle_get_cli_base()` — if no venv AND no global `fork` → returns venv path anyway (soft fallback)

The generated `done-helper.sh` and `lifecycle-pre.sh` use `lifecycle_get_cli_base` output. If the path doesn't exist, these scripts silently fail (`|| true`), which is correct for graceful degradation but means the DB never gets lifecycle updates.

**Impact:** In edge case (no venv, no global fork), lifecycle transitions silently skipped. Launches stay in intermediate state.

**Recommendation:** Either make both functions symmetric (same fallback), or add a pre-flight check in `spawn.sh` that refuses to launch if `lifecycle_get_cli_base` returns a non-existent path.

---

### F3: `reconcile_expired_leases` has no caller in the skill ⚠️ MEDIUM

**Severity:** MEDIUM
**Confidence:** high (grep confirmed empty)
**Heuristic:** H5 — Authority depends on evidence (lease expiry) but no one checks

**Evidence:**
- `reconcile_expired_leases()` exists in the service but is never called by the skill or any CLI command exposed to the skill.
- Only callable via Python import or `summary`/`list-quarantined` (read-only).
- The previous autoresearch session called it manually to clean 72 stale records.

**Impact:** Over time, DB accumulates stale ACTIVE/SPAWNING launches that never get cleaned. Only fix is manual intervention.

**Recommendation:** Add a `reconcile` CLI subcommand OR call reconcile automatically in `summary`/`list-active`/`init`. Alternatively, add a cron-like call in `cmd_init`.

---

### F4: `/tmp/fork-live-registry/` and `/tmp/fork-live-done/` — legacy split-brain INFO

**Severity:** LOW
**Confidence:** high (code trace)
**Heuristic:** H4 — Two file locations for similar state

**Evidence:**
- `kill-all` cleans `REG_DIR` files but NOT `/tmp/fork-live-registry/` or `/tmp/fork-live-done/`.
- These appear to be from the pre-decomposition monolith. The monolith wrote to both locations.

**Impact:** After `kill-all`, `/tmp/fork-live-registry/` still has stale `.meta` files. If any code reads from this location, it gets stale data.

**Recommendation:** Clean up in `kill-all` or deprecate `/tmp/fork-live-registry/` entirely if `REG_DIR` is the canonical location.

---

### F5: Done-helper error message is static ⚠️ LOW

**Severity:** LOW
**Confidence:** high (code inspection)
**Heuristic:** H5 — Evidence quality

**Evidence:**
```bash
lifecycle_mark_failed="${_cli} mark-failed --launch-id ${launch_id} --error 'Agent exited with non-zero code' --json"
```

The error message is always `'Agent exited with non-zero code'` regardless of actual exit code. The exit code IS captured in `.done` and `.meta` files but NOT passed to `mark-failed`.

**Impact:** DB records show generic error instead of actual exit code. Debugging requires cross-referencing `.done` file.

**Recommendation:** Pass `$_ec` to `--error "Agent exited with code $_ec"`.

---

### F6: `lifecycle_request` returns empty string on failure (not error code) INFO

**Severity:** INFO
**Confidence:** high

**Evidence:** When `_call_fork_launch` fails (no venv, no fork, timeout), `lifecycle_request` does `|| true` and returns empty string. Caller (`cmd_launch`) uses `|| return 0` which means launch proceeds WITHOUT tracking.

**Verdict:** This is by design (ADR-002: graceful fallback). The skill degrades gracefully when the repo CLI is unavailable.
